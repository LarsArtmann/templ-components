Verdict

The codebase is architecturally sound and the library's type model is internally consistent — but I shipped a release with no changelog, frozen docs, and a few factual lies in AGENTS.md. The two real type-safety gaps (unvalidated ButtonProps.Type and FormMethod) were the only deviations from the library's own enum convention. All safe-to-fix items are now done; the rest is blocked on the cqrs-htmx repo.

01Honest Answers

Did I lie to you? Yes.

Three factual falsehoods were sitting in the docs I authored: (1) AGENTS.md claimed "SanitizeID not used internally" — it is, in forms/radio_go.go:55. (2) AGENTS.md said "46 generated files" — the real count is 51. (3) A status report declared "adminui Tailwind infrastructure ready" while the compiled admin-tw.css was never added to assets.go's go:embed — so it is never served. I declared victory prematurely.

What did I forget?

Shipped v0.4.0 with no CHANGELOG entry (the [Unreleased] section still said "Nothing yet"). FEATURES.md was frozen at v0.3.0 — every metric wrong. CONTEXT.md and TODO_LIST.md were dated 2026-06-20. The demo hardcodes the version string and showcases only ~48 of 73 components.

What's stupid that we do anyway?

Component and icon counts are hand-maintained in four separate docs (README, FEATURES, AGENTS, CONTEXT) that drift the moment a component ships. README even contradicted itself — the forms header said "12" while the total of 73 only adds up with forms=16. This is a split brain by design.

Ghost systems?

Yes. admin-tw.css is a ghost artifact — compiled, committed, but never embedded or served. And the status report described templ-components-colors.css as a current deliverable after it was deleted — a self-contradiction within the docs corpus.
