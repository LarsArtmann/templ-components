!Open Findings

Issues identified but NOT fixed — they require API changes, larger refactors, or design decisions beyond a review pass.

High Impact

FormFieldWrapper not used by Input, Select, Textarea, Radio

forms/input.templ:78, forms/select.templ:54, forms/textarea.templ:41

FormFieldWrapper exists specifically to DRY up the Label + children + FieldError + helpText pattern. Only FileInput uses it. Input, Select, Textarea, and RadioGroup all manually repeat the same 3-line pattern. This is the exact duplication the wrapper was created to prevent.

Why not fixed here Refactoring changes generated HTML structure for 4 components. Requires golden file regeneration and may affect consumers who assert on specific markup order. Should be a dedicated refactor PR.
High Impact

Modal and Drawer JS is ~90% duplicated

display/modal.templ:57-109 vs display/drawer.templ:63-121

The open/close functions, focus trap (Tab key handling), focus save/restore via data-tc-prev-focus, and the focusable element selector string are nearly identical between Modal and Drawer. The selector 'a[href], button:not([disabled]), textarea, input, select, [tabindex]:not([tabindex="-1"])' appears 4 times across these two files.

Recommended approach Extract a shared overlayScript(componentName, id, transformDirection) function that generates the common JS. Or extract the focusable selector to a shared Go string constant.
Medium

Modal/Drawer inline onclick handlers (CSP)

display/modal.templ:31,45 — display/drawer.templ:31,51

The backdrop and close button use onclick={ modalCloseHandler(props.ID) } which renders as an inline onclick="tcCloseModal('id')" attribute. This violates strict CSP (which the library otherwise enforces via nonce'd scripts).

Fix pattern (already used by Alert/Accordion/Dropdown) Replace with data-close="modal" attribute + global click listener in the nonce'd script. The dismiss pattern in utils.DismissScript() shows the exact approach.
Medium

Form.Content prop instead of children

forms/form.templ:21

FormProps.Content templ.Component requires consumers to pass content as a prop. Every other wrapper component (Card, Modal, Drawer, InputGroup, FormFieldWrapper) uses { children... }. This breaks the composition pattern and makes Form harder to use than it should be.

Medium

Alert vs Toast: inconsistent icons and opacity for same severity

feedback/alert.templ:52-57 vs feedback/toast.templ:185-190

Same FeedbackType renders different icons in Alert vs Toast:

Severity Alert Icon Toast Icon
Error X ExclamationTriangle
Success Check CheckCircle

Also: dark mode bg opacity differs (/20 for Alert, /30 for Toast). Same severity should produce the same visual treatment.

Low

errorpage htmlEscape reimplements stdlib

errorpage/handler.go:352-371

Hand-rolled HTML escaper duplicates html.EscapeString() from the stdlib. The only difference is escaping ' → ', which stdlib also does in modern Go. Could be replaced with html.EscapeString(title).

Low

errorpage hand-rolled HTML shell via string concatenation

errorpage/handler.go:331-349

renderWithShell() builds an HTML document via fmt.Fprint string concatenation. Could be a templ component for consistency and safer escaping. Acceptable for an error fallback path.
