02Important Issues

Type Safety

Size lookups use map[string]string instead of typed keys

display/modal_go.go:36 · display/drawer_go.go:51

modalSizeLookup and drawerSizeLookup are declared as map[string]string with string(ModalSizeSM) conversions at every entry and call site. Every other lookup table in the codebase (buttonVariantLookup, feedbackStyleMap, familyStyleMap) correctly uses typed keys. This is inconsistent and loses compile-time safety — a typo in a size string would silently fall through to the default.

Fix Change to map[ModalSize]string and map[DrawerSize]string. Remove all string() conversions. The utils.Lookup[K,V] generic already supports this.
Split Brain

44 files of import-grouping churn from generator version mismatch

go.mod: templ v0.3.1020 · installed: v0.3.1036

The installed templ generator (v0.3.1036) groups templ and templruntime imports into the main import block, while the committed files (generated by v0.3.1020) keep them in a separate block. Every templ generate run produces a 44-file diff of pure import reordering. This pollutes PR diffs and git history with noise.

Decision needed Either pin the generator version to match go.mod, or accept the new grouping as the standard and commit the normalization once. The recent commit history already shows 3 "regenerate: normalize import grouping" commits — this churn is recurring.

04Type Model Analysis

What works well

The BaseProps + ComponentProps interface with GetBaseProps()/SetBaseProps() method promotion is a solid composition pattern. The utils.Lookup[K,V] generic (recently refactored from MapEnum) is the right abstraction for enum-to-class mappings. The FeedbackType unification with backward-compatible aliases (AlertType = FeedbackType) is well done.

What needs work

Stringly-typed lookups: modalSizeLookup and drawerSizeLookup use map[string]string with string() conversions, while every other lookup uses typed keys. This is the one remaining type safety gap in the enum system.

Overlay panel config split brain: overlayPanelConfig uses openClasses string / closeClasses string containing comma-separated JS class strings ("'scale-100', 'opacity-100'"). These are JS-source-as-Go-strings — not type-safe, not validated. A typo produces silent runtime breakage.

Overlay config: before / after

Current (stringly-typed JS fragments):

type overlayPanelConfig struct {
    openClasses  string // "'scale-100', 'opacity-100'"
    closeClasses string // "'scale-95', 'opacity-0'"
}

Better (typed slices, join at use site):

type overlayPanelConfig struct {
    openClasses  []string // {"scale-100", "opacity-100"}
    closeClasses []string
}
