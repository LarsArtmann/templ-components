| go.mod files | 1 (single module) |

## Why Modularization Is NOT Recommended

### 1. Library, Not Application

This is a **publishable Go library** consumed via `go get`. Multi-module repos add complexity for consumers:
- Must understand which sub-module to import
- Version pins become complicated (`display/v1.2.3` vs `feedback/v1.1.0`)
- Coordinated releases become harder

### 2. Small Codebase (~18K lines)

18K lines across 9 packages is well within single-module manageability. Go's package system already provides the namespace separation needed. Multi-module is typically valuable at 50K+ lines or when different teams own different modules.

### 3. Tight Coupling Would Remain

The import graph shows `utils` is imported by everything, and `icons` by 5/8 packages. Splitting into sub-modules would create:
- Complex `replace` directives during development
- Version coordination nightmares (a `utils` change affects all)
- Higher probability of version skew

### 4. Two External Dependencies

With only `templ` and `tailwind-merge-go`, there's no dependency isolation benefit. Every sub-module would list the same two deps.

### 5. Single Version Story

As one module, consumers get a consistent API surface. `go get github.com/larsartmann/templ-components@v0.2.0` gives everything at once. With sub-modules, consumers must coordinate versions themselves.

## What TO Do Instead

1. **Use Go packages (already doing this)** — Package boundaries provide compile-time isolation
2. **Keep `internal/svg` internal** — Already correctly prevents external imports
3. **Consider `internal/js` for shared JS runtime** — If JS complexity grows
4. **Keep all `*_templ.go` committed** — Already correct for a library

## If The Project Grows

If the codebase exceeds 50K lines or specific packages need independent versioning:

| Sub-module | Contents | When |
|------------|----------|------|
| `utils` | Base types, helpers | If other projects need these |
| `icons` | Icon system | If icon count exceeds 100+ |
| `core` | display + feedback + forms | If layout/htmx become optional |

But for now: **one module, 9 packages, clean DAG. This is the right structure.**
