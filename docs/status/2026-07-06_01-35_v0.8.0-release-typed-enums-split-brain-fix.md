## D) TOTALLY FUCKED UP ЁЯТе

### 1. Multi-Module Documentation Lie (CRITICAL тАФ pre-existing, found this session)

**AGENTS.md, flake.nix, and CHANGELOG v0.7.0 all described a 6-module Go workspace that never existed on master.** The modularization was done on `modularize/strategic-split` (12 commits, pushed to origin) but never merged. The v0.7.0 release commit (`2b57bd0`) shipped the false claim "split into 6 Go modules coordinated by go.work" тАФ consumers reading the CHANGELOG would expect sub-modules that don't exist.

**Impact:**

- `golangci-lint run ./svg/...` in flake.nix errored (`lstat ./svg/: no such file or directory`) тАФ the lint command was broken.
- AGENTS.md build commands referenced `go.work`, sub-module iteration, and `GOWORK=off` isolation tests тАФ none of which work on single-module master.
- I wrote multi-module docs for README.md, CONTRIBUTING.md, and icons-only-adoption.md based on the false AGENTS.md тАФ had to revert all 4 files.
- ~30% of this session's time was spent discovering and correcting this lie.

**Root cause:** The modularization report (`docs/status/2026-07-05_20-39_modularization-status-report.md`) said "NOT merged тАФ waiting for review" but the subsequent session (session 9/v0.7.0 release) committed the multi-module claims to AGENTS.md and CHANGELOG as if the merge had happened. Nobody verified.

**Fixed:** a0dbae7 corrected all three files. CHANGELOG v0.7.0 now says "prototyped, not shipped."

### 2. Accordion BuildFlow Corruption

BuildFlow's pre-commit `templ-gmt` step rewrote `display/accordion.templ` to use `utils.Class("overflow-hidden", transitionNormal, templ.KV(...))` тАФ but `templ.KV` returns `KeyValue[string, bool]` which cannot be passed to `utils.Class(...string)`. This caused a compile failure. Reverted the accordion change; the motion-reduce addition to accordion is still missing.

### 3. BuildFlow Golden File Drift

BuildFlow's pre-commit adds `motion-reduce:*` classes to source `.templ` files, which changes rendered HTML, which breaks golden tests. This happened on 3 separate commits this session. Each time required `-update` to regenerate golden files. The pattern: commit тЖТ BuildFlow modifies source тЖТ golden test fails тЖТ fix тЖТ re-commit. This is a fundamental friction point.

## E) WHAT WE SHOULD IMPROVE ЁЯФз

### Process

1. **Verify doc claims before committing** тАФ the multi-module lie persisted across 2 releases because nobody ran `ls go.work` or `ls svg/go.mod` against the claims in AGENTS.md. Rule: any architectural claim in docs must be verified against the actual filesystem before committing.
2. **Add a "doc reality" CI check** тАФ script that verifies AGENTS.md claims (module count, package list, lint paths) match reality. Prevents future split brains.
3. **Commit incrementally** тАФ this session started by batching 8+ changes without committing. When BuildFlow corrupted files, recovery was harder. Rule: commit after each self-contained change.
4. **Check `git diff` before `templ generate`** тАФ BuildFlow modifies `.templ` source files in the background. Always check `git diff` before regenerating to catch unauthorized changes.
5. **BuildFlow golden-file workflow** тАФ BuildFlow adds motion-reduce classes that change golden output. Need either: (a) a pre-commit golden update step, or (b) accept that golden files always need `-update` after BuildFlow runs.

### Code Quality

6. **Accordion motion-reduce still missing** тАФ the `transition-all duration-200` on the accordion panel still lacks `motion-reduce` fallback. BuildFlow tried to fix it but broke compilation. Needs manual fix using a different approach (inline `motion-reduce:` classes, not `transitionNormal` constant + `templ.KV`).
7. **Stale remote branch** тАФ `origin/modularize/strategic-split` still exists. Decide: delete or keep for future reference.
8. **Tag signing** тАФ configure `gpg.ssh.allowedSignersFile` so tags can be SSH-signed per the release convention.

## G) Top #1 Question I Cannot Answer

**Should the `modularize/strategic-split` branch be merged to master, or abandoned and deleted?**

The branch has 12 commits implementing the 6-module split. It was never merged because:

- The session report said "waiting for review/decision"
- The user never explicitly asked to merge it
- v0.7.0 was released without it (but with false claims about it)

The dilemma:

- **Merging now** means all the multi-module docs (which I just corrected to single-module) would need to be re-corrected back to multi-module. The release script, CI workflow, and flake.nix were updated on the branch for multi-module support. But the BuildFlow branch-switching chaos documented in the modularization report makes the branch potentially unstable.
- **Abandoning it** means deleting 12 commits of work. The split may genuinely benefit consumers who only want `icons` without pulling `tailwind-merge-go`. But git analysis showed 80%+ co-change, suggesting the packages aren't independent enough to justify separate modules yet.
- **Deferring** keeps the branch alive as a reference but leaves the repo in a state where the "ideal" architecture and the "actual" architecture diverge.

I lean toward **deferring to post-v1.0** тАФ the split adds complexity (replace directives, per-module tags, go.work management) without clear consumer demand yet. The icons-only adoption path already works within the single module. But this is a product direction question, not a technical one.
