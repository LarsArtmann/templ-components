7. **Applied alejandra formatting to `flake.nix`.** Function-arg destructuring `outputs = inputs @ { self, flake-parts, ... }: flake-parts.lib.mkFlake {inherit inputs;} {` is the canonical alejandra form.

## e) WHAT WE SHOULD IMPROVE 💡

### Architecture / design

1. **Stronger versioning discipline.** The `Version` constant + drift-guard test is excellent. The missing piece is a Makefile / script that does `bump-version → update CHANGELOG → tag → push` atomically, so a release is one command. Currently it's 3-4 manual steps with the BuildFlow hook in the middle, which is exactly how this session went sideways twice (the first commit attempt was rejected by the pre-commit hook).

2. **`utils.Class()` lock contention.** AGENTS.md notes it uses a `sync.Mutex` to protect `tailwind-merge-go`'s LRU cache. If benchmarks under load ever show it as a hotspot, swap to a per-shard cache or a `sync.Map`. Not measured yet.

3. **ComponentProps interface contract test.** The "all 25+ props structs auto-satisfy `utils.ComponentProps`" claim in AGENTS.md is convention-enforced, not test-enforced. A simple `var _ utils.ComponentProps = (*ButtonProps)(nil)` test in each component package would catch a future "whoops I forgot to embed BaseProps" bug at compile time. Low cost, high signal.

4. **Golden test baseline refresh.** The `internal/golden` test infrastructure is solid (post-isolation fix), but a deliberate `make refresh-goldens` command would let consumers regenerate after a deliberate CSS class change without manually running `-update`.

5. **Release notes generation.** When a tag is cut, auto-generate a release-notes Markdown from the `[Unreleased]` → version block transition. Right now it's manual.

### Process

6. **Pre-commit hook transparency.** BuildFlow emits lots of build flow UI but the actual _failure reason_ (e.g. "flake.nix: nixfmt not formatted") often scrolls past the noise. A `--verbose` mode or a "last 20 lines of failure" tail would save debugging time. (This session lost 5+ minutes to figuring out the `nix-fmt` failure root cause.)

7. **The templ version pin / system binary drift.** Either commit to a `flake.nix` `templ` derivation as the canonical source (so `nix develop` provides `v0.3.1036` matching `go.mod`) or document the expected mismatch in AGENTS.md. Currently the relationship between the two is implicit.

8. **More `flake.nix` polish.** It's still missing `checks.x86_64-linux.<name>` (run `nix flake check` to run tests in a sandbox) and a `treefmt-nix` step. The LarsArtmann standard is moving toward flake-parts + treefmt-nix; this flake is flake-parts + raw attributes. ADR-worthy.

### Library completeness (gaps vs the ideal v1.0)

9. **No `Combobox` server-side hookup** in the current `forms.Combobox` — it's pure client-side. A future `ComboboxProps.SourceURL` for HTMX-driven option lists would close a real-world gap.

10. **No `Drawer` for left-side navigation use case** in the existing demo. The component supports both `left` and `right` positions but the demo only shows one.

11. **No data-table component.** `display.Table` is presentational only. A `display.DataTable` with sorting/filtering/pagination would be a high-value addition for admin UIs (which is exactly the consumer that cqrs-htmx/adminui needs).

## g) My Top #1 question I cannot figure out myself

**Should `utils.Version` and the Git tag be cut on the same commit, or should `Version` be bumped in a separate dedicated commit (release-prep) for clean `git diff v0.5.0..v0.6.0` reviewability?**

In this session, `Version = "0.6.0"` and all the 0.6.0 work landed in the same commit (`f93a801`). The upside: one atomic release commit. The downside: `git diff v0.5.0..v0.6.0 -- utils/version.go` is part of a 50-file diff and the version bump isn't reviewable in isolation. Many release-engineering best-practices recommend a two-commit pattern:

1. **Commit 1 (release-prep):** just `utils/version.go` + `CHANGELOG.md` (bump + heading transition).
2. **Commit 2 (release):** the actual content/feature commits that belong in the release.

Then `git tag v0.6.0` lands on commit 2 (or on a commit 3 that's empty), and `git diff v0.5.0..v0.6.0` cleanly shows the code changes, while commit 1 is a tiny atomic version-bump.

I don't have enough context on LarsArtmann's house style for release commits to decide. Looking at the existing tags (`v0.4.0`, `v0.5.0`), it appears the convention has been **one release commit per version** (the v0.5.0 commit `e72338c` contains the version bump + 47-file diff), which matches what we just did. So we probably already followed convention.

But for v1.0 — when the diff will be larger and a clean release is more ceremonious — I'd want to confirm: **one-commit release, or two-commit (prep + release)?**
