## E) WHAT WE SHOULD IMPROVE 🎯

### Process improvements

1. **Run dark-mode + motion-reduce + RTL tests BEFORE committing new components**, not "later." We got lucky — all classes were present — but we didn't verify.
2. **Generate golden files for ALL new display components.** ContextMenu and Carousel are missing golden files. This is inconsistent.
3. **Register new JS-bearing components in `integration/csp_nonce_test.go`.** TagsInput, ContextMenu, and Carousel all have inline scripts but weren't added to the CSP integration test.
4. **Add new components to the demo binary.** 5 of 9 new components are invisible in the demo.
5. **Don't promise 80% coverage.** The templ runtime generates branch-heavy code. Set realistic targets (75% for packages with heavy generated code).
6. **Write the example_test.go in external package style** (`display_test`) for godoc rendering — this is the pattern the library established.
7. **Run `TestMotionReduceCompliance` explicitly** after adding transitions/animations to new components.

### Code improvements

8. **ContextMenu uses `classList.add('hidden')` but Tailwind v4 may purge `hidden` if it's not in a scanned file.** Should verify the `hidden` class is available or use inline `style.display = 'none'`.
9. **Carousel slide labels use `fmt.Sprintf` inside the template loop** — this works but generates non-deterministic line numbers in errors. Acceptable trade-off.
10. **Calendar `calendarReplacePlaceholders` uses `strings.NewReplacer`** — correct but allocates per call. For a calendar rendered once per page load, this is fine.
11. **TagsInput JS creates DOM elements with `innerHTML` for the SVG** — this is CSP-compliant because the script has a nonce, but it's the kind of thing that could break if the CSP is tightened to disallow `unsafe-inline` on scripts. The nonce covers this, but worth noting.
12. **HoverCard has no `Content` fallback when nil** — renders an empty tooltip div. This is graceful but could confuse screen readers announcing an empty tooltip.

## G) Top 2 Questions

### 1. Should we push v0.17.0 to remote now, or wait until the CSP/RTL/dark-mode compliance gaps are verified?

The release is committed and tagged but NOT pushed. The compliance tests (dark-mode, motion-reduce, RTL, CSP nonce) were not explicitly run against the 5 newest components. If any of them fail, we'd need a patch release. Should we:

- **(a)** Push now and patch if needed (fast feedback from consumers), or
- **(b)** Verify compliance first, patch if needed, then push a clean v0.17.0?

### 2. Should the coverage target be revised from 80% to 75% for templ-heavy packages?

The 80% target was set in the Pareto plan but proved unrealistic for packages where templ generates branch-heavy buffer/context management code. The generated `*_templ.go` files have 15-20% unreachable branches (context cancellation, buffer release error paths) that would require white-box testing of generated code. Options:

- **(a)** Lower the target to 75% for packages with heavy templ code (display, forms, feedback, navigation), keep 80% for lighter packages (utils, htmx, icons, internal)
- **(b)** Keep 80% and invest in white-box coverage of generated code
- **(c)** Exclude generated `*_templ.go` from coverage (add `//go:build coverage_exclude` tag or use `-coverage-exclude` in golangci-lint config)
