## e) WHAT WE SHOULD IMPROVE 🎯

### Architecture improvements

1. **Extract overlay overlay behavior to a shared Go function** — Modal and Drawer share: open/close class toggling, focus trap (Tab key handling), focus save/restore, focusable element selection. This should be `overlayScript(componentName, id, direction)` in `display/shared.go`.

2. **Consolidate CSP inline scripts** — 10+ components emit inline `<script nonce>` blocks. The global singleton guard pattern (`window.tc*Attached`) is repeated everywhere. Could extract a registration helper.

3. **Form field type hierarchy** — Every form field props struct repeats `Label string`, `Required bool`, `Error string`, `HelpText string`. An embedded `FormFieldBase` struct would DRY this at the type level while keeping backward compatibility via field promotion:
   ```go
   type FormFieldBase struct {
       Label    string
       Required bool
       Error    string
       HelpText string
   }
   ```

### Testing improvements

4. **Move from snapshot to golden for all packages** — Only `feedback/` uses golden files. The pattern in `internal/golden` is excellent (CSS class normalization, `-update` flag). Should be adopted across all packages.

5. **Add a nonce-presence assertion test** — A test that renders every component with a nonce and asserts `nonce=` appears in every `<script>` tag. Would prevent CSP regressions.

### Documentation improvements

6. **Update TODO_LIST.md** — It's stale. 6+ items completed this session aren't marked done. Should be updated immediately after each session.

7. **CHANGELOG.md** — The current session's changes (Alert/Toast unification, Form children breaking change, Pagination uint breaking change) should be documented for the next release.

### Type safety improvements

8. **More uint usage** — `ProgressBarProps.Current`/`Total`, `SpinnerProps.MaxLength`, `TextareaProps.Rows`/`MaxLength`, `InputProps.MaxLength` could all be uint. Same "make invalid states unrepresentable" principle.

9. **Enum validation consistency** — Currently 2 panic-on-unknown (InputType, icons.Name) and 10 map+fallback. The panic approach catches bugs earlier but is inconsistent. Should pick one pattern and document it.

### Library adoption of established tools

10. **Consider `github.com/nicksnyder/go-i18n`** for the pluralization problem (currently hand-rolled for ValidationSummary). Low priority — only one string needs it.

## g) TOP QUESTION I CANNOT FIGURE OUT MYSELF 🤔

**#1: Should we break the Modal/Drawer inline onclick CSP pattern now, or defer it?**

The Modal and Drawer currently use `onclick={ modalCloseHandler(props.ID) }` which templ renders as `onclick="tcCloseModal('id')"`. This is an inline event handler that violates strict Content Security Policy.

**The problem:** Replacing this with `data-close` attribute + event delegation changes the DOM structure slightly and requires regenerating the close handler script. It's not hard, but it's a behavior change that consumers might notice if they rely on the specific `onclick` attribute.

**What I can't decide:** Is the CSP compliance worth the breaking change NOW (before v1.0 freeze), or should we document it as a known limitation and fix it in v1.0? The library already states "all inline scripts use nonce" in AGENTS.md — but these onclick handlers technically aren't `<script>` tags, they're HTML attributes. Different CSP directive (`script-src-attr` vs `script-src`).

**Context I need:** Does the target audience (Go + HTMX developers) actually use strict CSP with `script-src-attr 'none'`? If yes, this is critical. If most consumers use `script-src 'self' 'nonce-...'` without the attr restriction, it's lower priority.

**What I would do with the answer:** If CSP-attr is common → fix immediately (45min). If not → document as v1.0 scope item and move on.
