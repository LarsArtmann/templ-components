   I made changes and didn't log them. This is the #1 miss.

2. **Decision-testability pattern** — When documenting an architectural
   decision in code comments (stay/move thresholds, field-count thresholds),
   consider adding a compile-time or runtime guard that catches violations.
   Not always worth it, but should be considered.

3. **Cross-reference check** — After resolving items from a follow-up list,
   verify the source list (status report, TODO_LIST.md, etc.) is updated to
   reflect the resolution. The dedup-sprint status report at
   `2026-07-06_06-24_dedup-extraction-sprint.md` still lists items 6 and 7
   under "Short-term (next session)" — they're now resolved but that file
   doesn't know it.

### Design

4. **The decisions themselves are correct.** Both items were conditional
   triggers that haven't fired. Applying YAGNI (don't move goBackScript
   prematurely, don't restructure overlayShellProps prematurely) is the right
   call. The DismissButton precedent is a strong anchor — it demonstrates the
   library already has a proven promotion path when the trigger fires.

5. **The documentation is actionable.** Both comments name the exact trigger
   condition and the exact remediation step. A future developer hitting the
   trigger doesn't need to re-derive the decision.
