1. How scalable & modular is the current architecture?

Verdict: highly modular and scalable. The library is a single Go module organized into 13 packages with a strictly layered, acyclic dependency graph. There are no god-objects, no circular imports, and the external dependency surface is deliberately minimal (three repos). For a server-rendered component library this is the right shape.

Wins

• Layered & acyclic. utils is a true leaf; nothing depends upward into feature packages. Import order is deterministic.

• Tiny, intentional dependency surface. 3 external repos — templ (rendering), tailwind-merge-go (class dedup), go-error-family (error classification). No HTTP framework, no ORM, no logging lib. Zero supply-chain risk surface.

• icons-only adoption is emergent. icons transitively depends only on internal/svg + templ + utils — no CSS framework, no tailwind-merge-go. Consumers can adopt just the icon system without the rest (see docs/icons-only-adoption.md).

• No split brains. Each concept lives in one package; shared SVG paths have a single source of truth in internal/svg.

Watch items (non-blocking)

• display is the heaviest package (~12.5k LOC, 30 components, fan-out 5). It is cohesive — every member is a presentational component — but it is the natural seam if the library ever needs a finer split.

• utils is a 204-edge hub. Acceptable because it is a thin leaf (BaseProps, Class(), Lookup, EnsureID, motion constants), not a god-package. But any new global added there is immediately inherited by every consumer — keep it deliberately small.

2. Composability & service-orientation

Service-orientation does not map onto a UI library — there are no services. The directly analogous property is composability, and here the library is strong.

Composability seams

Seam Mechanism Where it shows up
Uniform props contract utils.ComponentProps interface + BaseProps embedding All 26+ props structs auto-satisfy via pointer-receiver promotion
Render slots templ.Component fields on props Card.Body, Card.Header, Table.Body, SimpleCard.Body, PageHeader.Breadcrumb/Action, SidebarNav.Brand/Footer
Children pass-through { children... } Grid, Form.Content, composition tests in integration/
Typed enum extension map[EnumType]string + utils.Lookup 37 closed-set enums, all with IsValid(); fallback-safe
HTMX as opt-in layer typed HxGet/HxTarget/HxSwap fields, omitted when empty StatCard, LoadMore, Form, FilterDropdown — works without HTMX present
CSS theming without Go changes Tailwind v4 @theme override templ-components-theme.css semantic aliases; --color-blue-600
Why this composes well

Every component speaks the same contract (BaseProps → Class/Attrs/ID/AriaLabel/Nonce propagation), so components nest arbitrarily and consumer overrides always win via utils.Class() tailwind-merge resolution. Slots (templ.Component) are the primary composition primitive — no framework lock-in, no runtime DI.

Recommendations & roadmap

  1. Hold the module split until v1.0. A full per-package go.mod + go.work split was prototyped on modularize/strategic-split and analyzed in docs/modularization/PROPOSAL.md. Benefit: selective imports, enforced boundaries, independent versioning. Cost: 10 go.mod files to keep in lockstep, consumer friction, tooling overhead. The current single-module shape is correct while the API is still pre-1.0. Re-evaluate at v1.0.
  2. Keep utils on a diet. Its 204 in-edges make it the highest-leverage package in the library. Any new export is a commitment. Prefer a feature-local helper unless ≥2 packages need it (the existing ADR 0010 sub-template extraction rule generalizes here).
  3. Watch display for a future split. If it grows past ~40 components, consider extracting display/overlay (Modal, Drawer, Dropdown, Popover, ContextMenu, HoverCard, Tooltip) and display/data (Table, DataTable, DefinitionList, Grid) into siblings. Not needed today.
  4. Preserve the icons-only property. The fact that icons carries no CSS/Tailwind dependency is a real adoption path. Do not let icons grow a dependency on display or tailwind-merge-go.
  5. Document the composition contract in one place. The BaseProps/slot/enum patterns are described across AGENTS.md and ADRs. A single docs/composition.md (or a recipes entry) would make the composability story discoverable for new consumers.
